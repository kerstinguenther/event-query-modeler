What is eqmn.io?

eqmn.io is a modeling tool based on Camundas bpmn.io. It can be used to model EQMN models.


What is EQMN?

EQMN is the abbreviation for Event Query Model and Notation and defines the grammar and notation for a modeling language to model event queries. Following, the elements of EQMN are described briefly and an example EQMN model is shown to give you a clue how such a model should look like.


InputEvent



An input event is the starting point of an EQMN model. There must be at least one input event, but there is no upper limit. Each input event must be named with an (existing) event type.

Input events can be placed inside a window or an interval.


Input events can have one of the following incoming connections:

  • loose sequence from input event

  • loose sequence from operator

  • loose sequence from interval


Input events must have one of the following outgoing connections and optionally an association to a condition:

  • loose sequence to input event

  • loose sequence to operator

  • loose sequence to interval

  • sequence flow to operator

  • sequence flow to output event


OutputEvent



An output event is the end point of each EQMN model. There must be exactly one output event and it can be optionally named.



Output events must have one of the following incoming connections:

  • sequence flow from input event

  • sequence flow from operator

  • sequence flow from interval

  • sequence flow from window


Operators



ConjunctionOperator connects multiple EQMN elements (like input events, other operators or intervals) with a logical AND, i.e. all incoming elements must be true.





DisjunctionOperator connects multiple EQMN elements (like input events, other operators or intervals) with a logical OR, i.e. one incoming element must be true.




Conjunction and disjunction operators must have at least two incoming sequence flows from one of the following:

  • input event

  • disjunction/conjunction/negation operator

  • interval




NegationOperator negates one EQMN element (like input event, other operator or interval) with a logical NOT, i.e. the incoming element must be false.



Negation operators must have exactly one incoming sequence flow from one of the following:

  • input event

  • disjunction/conjunction/negation operator

  • interval


Conjunction, disjunction and negation operators can have additionally one of the following incoming connections:

  • loose sequence from input event

  • loose sequence from conjunction/disjunction/negation operator

  • loose sequence from interval


Conjunction, disjunction and negation operators must have one of the following outgoing connections:

  • loose sequence to input event

  • loose sequence to conjunction/disjunction/negation operator

  • loose sequence to interval

  • sequence flow to conjunction/disjunction/negation operator

  • sequence flow to output event




ListOperator connects multiple EQMN elements (like input events or windows) as input.



List operator must have at least one of the following incoming connections:

  • sequence flow from input event

  • sequence flow from window


List operator must have the following outgoing connection one time:

  • sequence flow to output event


Windows

All windows must contain exactly one input event. All windows except the empty one must be named with a time interval (e.g. '10 minutes') respectively a length definition (e.g. '20').



Window is an empty window that keeps all instances of the specified input event.





TimeWindow is a window that keeps all instances of the specified input event that occurred within the specified time interval.





LengthWindow is a window that keeps the last n instances of the specified input event. n must be specified.





SlidingTimeWindow is a window that keeps all instances of the specified input event that occurred within the specified time interval. If the specified time elapses, the window moves and the time interval starts again.





SlidingLengthWindow is a window that keeps the last n instances of the specified input event. n must be specified. If the window is full, it moves.





SlidingBatchTimeWindow is a window that keeps all instances of the specified input event that will occure within the specified time interval in the future. So it waits for incoming events. If the specified time elapses, the window moves and the time interval starts again.





SlidingBatchTimeWindow is a window that keeps the next n instances of the specified input event. n must be specified. So it waits for events. If the window is full, it moves.



All windows must not have an incoming connection.


All windows must have one of the following outgoing connections:

  • sequence flow to list operator

  • sequence flow to output event


Interval



Intervals can be used to define a time interval in which a complex event pattern should take place. Each interval must be named with a time interval (e.g. '10 minutes' or '1 hour'). An interval must contain at least one input event, but there is no upper limit.



Intervals can have one of the following incoming connections:

  • loose sequence from interval

  • loose sequence from input event

  • loose sequence from conjunction/disjunction/negation operator


Intervals must have one of the following outgoing connections:

  • loose sequence to interval

  • loose sequence to input event

  • loose sequence to conjunction/disjunction/negation operator

  • sequence flow to conjunction/disjunction/negation operator

  • sequence flow to output event


Condition



Conditions are textual annotations to restrict input events, output events or the complete input of the query by adding a condition to the sequence flow connecting an element to the output event.



Connections


Sequence is a connection to define the sequence flow of the model.




LooseSequence is a connection to define a temporal sequence between EQMN elements.



More features?

Next to the modeling, you can validate the event types and event attributes in your model by importing a predefined event hierarchy from a web service. Furthermore, you can translate the visual model to an event processing language – currently Espers EPL and Drools Rules Language (with Drools Fusion). Note that not all features of EQMN can be translated into each event processing language. The tool will let you know about validation errors.
